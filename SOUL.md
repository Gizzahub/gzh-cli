## gz는 무엇인가

`gz`는 개발자의 **반복 작업을 제거하는 통합 CLI**다. Git 플랫폼, IDE, 코드 품질,
패키지 매니저, 쉘·개발·네트워크·OS 환경 — 흩어진 도구들을 하나의 진입점으로 묶는다.

그러나 gz는 단일 프로그램이 아니다. **독립적으로 쓸 수 있는 라이브러리들의 연합**이며,
`gz` 바이너리는 그 연합의 가장 얇은 표면일 뿐이다.

그리고 gz는 **스위스칼이 아니다**. 어중간한 기능 열 개의 묶음이 아니라,
각자 제 영역에서 **가장 날카로운 전문 도구들**의 연합이다. gz가 어떤 영역을
맡기로 했다면, 그 영역의 작업은 기존 어떤 방법보다 편해야 한다.

## 왜 만드는가

- 개발자는 리포 1개가 아니라 **20개**를 다룬다. 도구들은 여전히 1개씩만 다룬다.
- 환경 전환(회사↔집, WiFi↔VPN, 노트북↔데스크탑)은 매일 일어나지만 자동화되어 있지 않다.
- 각 도구(git, brew, asdf, IDE)는 훌륭하다. 문제는 도구가 아니라 **도구 사이의 틈**이다.

gz는 그 틈을 메운다. 도구를 다시 만들지 않는다.

## 핵심 신념

### 1. 감싸되, 대체하지 않는다

git·brew·asdf·IDE가 진실의 원천이다. 우리는 편의를 더할 뿐 동작을 바꾸지 않는다.
사용자는 언제든 원래 도구로 돌아갈 수 있어야 한다.

- git을 직접 구현하지 않는다 — `git` CLI를 호출한다
- 패키지 매니저를 만들지 않는다 — 있는 것들을 통합 실행한다

### 2. 라이브러리가 먼저, CLI는 소비자

모든 기능은 독립 라이브러리(`gzh-cli-*`)로 먼저 존재한다.
CLI로 가능한 모든 작업은 라이브러리 import로도 가능해야 한다.

- 새 기능은 라이브러리 코드로 시작하고, CLI 통합(wrapper)이 뒤따른다
- wrapper 안에 핵심 로직이 들어가는 순간 이 신념은 깨진다

### 3. 대량 작업이 기본값이다 (Bulk-First)

단일 리포·단일 환경은 특수한 경우다. 기본 동작은 "전부 다"이며,
병렬 실행·진행 표시·dry-run이 표준 장비다.

### 4. 안전이 속도보다 먼저다

파괴적 작업은 명시적 동의를 요구한다. 기본값은 안전한 쪽이고,
위험한 쪽은 opt-in이다. 잃어버린 작업물은 느린 명령보다 나쁘다.

### 5. 구조도 철학이다 — 모노레포가 아닌 이유

12개 리포는 **독립 배포·독립 사용·독립 실패**를 위해 분리되어 있다.
한 라이브러리의 사용자는 다른 라이브러리의 의존성을 상속받지 않는다.
devbox는 이들을 조율할 뿐, 합치지 않는다.

## 제품의 경계

| 영역 | gz가 소유한다 | gz가 소유하지 않는다 |
|------|-------------|-------------------|
| Git | 대량 sync·clone·상태 진단, forge 통합 | Git 자체 구현, Git 서버, hooks 관리 |
| 환경 | 전환 자동화 (dev/net/os/shell) | OS 설정 도구의 재발명 |
| 품질 | 다중 언어 도구의 통합 실행 | linter/formatter 자체 개발 |
| UI | CLI + 단순 진행 표시 | GUI, 웹 인터페이스, IDE 플러그인, 고급 TUI |
| 자동화 | 파이프라인에서 쓸 명령·출력 제공 | CI/CD 오케스트레이션 자체 |

## 새 기능 승인 게이트

새 기능은 아래 4질문을 **모두** 통과해야 한다:

1. **틈을 메우는가?** — 기존 도구를 감싸 반복을 없애는가, 아니면 도구를 재발명하는가?
   (재발명이면 거절)
2. **라이브러리로 존재할 수 있는가?** — CLI 없이 import해서 쓸 수 있는 함수인가?
   (아니라면 설계를 다시)
3. **대량이거나 전환인가?** — 여러 리포·여러 환경에 걸친 작업인가, 단일 도구의
   고유 기능인가? (후자면 그 도구의 몫)
4. **가장 날카로워질 수 있는가?** — 이 영역에서 기존 도구·방법보다 확실히 더
   편해질 수 있는가? (어중간하게 남을 기능이면 거절)

## 신념이 충돌할 때

| 충돌 | 우선 | 이유 |
|------|------|------|
| 넓이 vs 깊이 | 깊이 | 어중간한 기능 열 개보다 가장 날카로운 기능 하나 |
| 안전 vs 속도 | 안전 | 잃어버린 작업물이 최악의 비용 |
| 단순함 vs 기능 | 단순함 | 적은 기능을 잘하는 것이 많은 기능을 어설프게 하는 것보다 낫다 |
| 라이브러리 vs CLI | 라이브러리 | CLI는 여러 소비자 중 하나일 뿐 |
| 일관성 vs 유연성 | 일관성 | 예측 가능한 동작이 사용자 실수를 줄인다 |
| 호환성 vs 혁신 | 호환성 | 깨지는 변경은 신뢰를 깬다 |

## 철학의 한 줄
